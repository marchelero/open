# Quick PRD Command

Generar un mini-PRD para: $ARGUMENTS

## Cuando usar

- Bug fix de 1-3 archivos maximo.
- Refactor chico sin cambio de comportamiento.
- One-liner con criterio de exito obvio.
- Ajustes de UX (color, copy, spacing).

## Cuando NO usar

- Feature nueva (usar `prd-agent` completo).
- Cambio de arquitectura.
- Cualquier cosa con >1 criterio de exito o >3 archivos afectados.
- Si dudas: usa `prd-agent` completo. Es mas barato que re-trabajar.

## Tu Task

NO invocar `prd-agent`. NO hacer intent map. NO hacer preguntas bloqueantes.

1. Escribir directamente `.opencode/prds/{YYYY-MM-DD_HHMM}-quick-{kebab-name}.prd.md` con este template:

```markdown
# {Quick title}

> Quick-PRD on YYYY-MM-DD_HHMM. Source: "{verbatim user request}"

## Status
DRAFT

## Context
[1-2 lineas: que esta mal / que se quiere cambiar]

## Objective
[1 oracion: que debe quedar distinto al final]

## Success Criteria

- [ ] {criterio 1 — observable y verificable}
- [ ] {criterio 2 si hay, max 3}

## Out of Scope
- [1 linea: que NO se toca]

## Files (expected)
- `path/to/file.ts` — [que cambio]
- `path/to/test.ts` — [test del fix]

---

*Generated by /quick-prd. Para tasks simples. Si crece en scope, regenerar con prd-agent.*
```

2. Reportar al usuario:
   - Path del mini-PRD.
   - Lista de archivos esperados.
   - Preguntar: "OK procedo a implementar? (s/n)"

3. Si confirma → implementar el fix.
4. Si el fix crece en scope durante implementacion (>3 archivos, >3 criterios):
   - Detener.
   - Notificar: "Esto ya no es un quick-PRD. Regenerando con prd-agent completo."
   - Crear PRD full y archivar el quick.

## Restricciones

- Max 3 success criteria.
- Max 3 archivos en "Files (expected)".
- Sin Asumptions (o 1 sola si es inevitable).
- Sin Risks table (el fix es chico, los riesgos se evaluan inline).
- Sin Delivery Milestones.
- Sin Open Questions (si hay una, no es quick).
