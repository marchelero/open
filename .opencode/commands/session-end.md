# Session End Command

Write a session snapshot to disk so the next session can resume with minimal context load. This is the WRITE half of the session memory system; `/session-start` is the READ half.

## Your Task

### Step 1 — Confirm with user

Ask briefly:
```
Voy a guardar snapshot de esta sesion. Resumen en 1-2 lineas primero:

  {one-line description of what you were working on}

¿Lo guardo como "{that description}" o queres otro titulo?
```

Wait for confirmation. If the user says "skip" or "no", exit without writing.

### Step 2 — Determine filename

```
DATE=$(date +%Y-%m-%d)  # or Get-Date -Format "yyyy-MM-dd" on Windows
SLUG=$(echo "{confirmed title}" | tr '[:upper:]' '[:lower:]' | sed 's/[^a-z0-9]/-/g' | cut -c1-50)
FILE=".agents/sessions/${DATE}-${SLUG}.md"
```

If that file already exists, append `-2`, `-3`, etc.

### Step 3 — Review what happened this session

Look back through the conversation. Identify (do not ask the user — infer from context):

- **Status** — what's DONE, IN PROGRESS, BLOCKED right now
- **Decisions made** — choices the user committed to
- **Files touched** — created / modified (with line counts if you can recall)
- **Open questions** — unresolved things that could affect next steps
- **Next steps** — what should happen in the next session
- **Commits** — `git log --oneline` since session start, or all commits in this branch

If the conversation is too short or vague to fill any section, write `TBD — no data this session` rather than inventing.

### Step 4 — Write the snapshot

Create `.agents/sessions/{DATE}-{SLUG}.md` with this template:

```markdown
# Session {DATE} — {title}

> Generated by /session-end. Read by /session-start on next session.

## Status

**DONE this session:**
- {item}
- {item}

**IN PROGRESS:**
- {item} — {where you got stuck, if any}

**BLOCKED:**
- {item} — {what's blocking}

## Decisions made
- {decision 1 — one line, with the reason if non-obvious}
- {decision 2}

## Files touched

| File | Change | Notes |
|---|---|---|
| {path} | created | {N lines, one-line purpose} |
| {path} | modified | {+N -N, what changed} |
| {path} | deleted | {why} |

## Open questions
- {question 1}
- {question 2}

## Next steps
1. {next concrete action}
2. {next concrete action}
3. {next concrete action}

## Commits this session
- `{short hash}` {commit message}
- `{short hash}` {commit message}

---

*Snapshot: {ISO timestamp} · Status: {DONE/PARTIAL/BLOCKED}*
```

### Step 5 — Update LATEST pointer

Copy the new snapshot to `.agents/sessions/LATEST.md` (overwrite if exists):

```bash
cp "{FILE}" ".agents/sessions/LATEST.md"
```

This makes the next `/session-start` find it without scanning by date.

### Step 6 — Report

```
Snapshot saved: .agents/sessions/{DATE}-{SLUG}.md
Pointer updated: .agents/sessions/LATEST.md

Status: {DONE | PARTIAL | BLOCKED}
Next step 1: {first item from Next steps section}

Maniana / cuando vuelvas:
  /session-start
```

## Behavior Notes

- **Opt-in discipline.** This command only runs when the user asks. The agent NEVER auto-snapshots mid-session (would add noise).
- **Cheap to write.** ~1-3K tokens for the snapshot file. No need to compress further.
- **Reviewed at next start.** `/session-start` reads the LATEST.md pointer, not all snapshots. Old snapshots are kept for archaeology.
- **Manual commit log.** The "Commits this session" section uses `git log`, not chat history. This keeps the snapshot truthful even if conversation is lost.

## When to Skip

- Trivial one-message sessions (single Q&A, no code changes)
- User explicitly says "skip snapshot" or "no guardes nada"
- You're in a `/verify` or `/checkpoint` flow (those have their own state-save commands)

## Integration

- `/session-start` — reads what this writes
- `intent-driven-development` skill — verify Next steps against ACs from PRD
- `git-workflow` skill — used to format the Commits section
